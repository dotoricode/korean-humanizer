## 사례 1. "왜 한국어 전용 기준이 필요한가" 1단락

**Before**

> 영어권의 AI 작성 징후는 흔히 장황한 안전문구, 과도한 전환어, 템플릿형 문장, 제목 대문자화, 이모지·대시 사용 같은 표면 신호로 요약되곤 한다. 그러나 이런 표면 징후는 한국어에 그대로 옮기기 어렵다. 한국어는 문말 종결어미가 화자 태도와 상대 높임을 직접 실현하고, 띄어쓰기에서도 허용 변이와 예외가 있으며, 쉼표 사용 밀도 역시 영어와 다르다. KatFish는 한국어의 **유연한 띄어쓰기**, **형태소 기반 구조**, **영어보다 덜 빈번한 쉼표 사용**을 핵심 차이로 지목했다. 국립국어원 자료 역시 상대 높임을 여섯 등급으로 설명하며, 해체·해요체의 구별과 종결어미 분포가 실제 문체 판단에 중요하다고 정리한다. 또한 맞춤법 규정은 단음절 연쇄와 보조 용언에서 붙여쓰기를 허용하는 구간을 분명히 제시한다. 즉, 한국어에서 "정답 하나"만을 기대하는 규칙은 실제 쓰기 관행을 오해할 수 있다.

**After**

> 영어권 AI 작성 징후는 보통 긴 안전 문구, 전환어 과다, 템플릿형 문장, 제목 대문자화, 이모지·대시 같은 표면 신호로 정리된다. 한국어로 그대로 옮기기는 어렵다. 한국어는 종결어미가 화자 태도와 상대 높임을 직접 드러내고, 띄어쓰기에 허용 변이가 있고, 쉼표 사용 밀도도 영어와 다르다. KatFish 는 한국어의 **유연한 띄어쓰기**, **형태소 기반 구조**, **영어보다 적은 쉼표 사용**을 핵심 차이로 짚었다. 국립국어원 자료도 상대 높임을 여섯 등급으로 설명하면서, 해체·해요체 구별과 종결어미 분포가 문체 판단에 중요하다고 본다. 맞춤법 규정도 단음절 연쇄와 보조 용언에서 붙여쓰기를 허용하는 구간을 분명히 둔다. 한국어에서 "정답 하나" 만을 기대하는 규칙은 실제 쓰기 관행을 잘못 읽기 쉽다.

## 사례 2. "한국어 관련 탐지 연구" 1단락 — 수치 보존 시연

연구 위키에서 가장 fact-heavy 한 단락. **숫자 / 모델명 / 벤치마크명** 이 humanize 후에도 글자 단위로 동일해야 한다.

**Before**

> 한국어 쪽에서는 최근 들어서야 의미 있는 축적이 나타났다. KatFish는 에세이, 시, 논문 초록 세 장르에서 인간/LLM 텍스트를 모은 최초의 한국어 탐지 벤치마크를 제시했고, 한국어 언어 특징에 기반한 KatFishNet이 기존 방법보다 평균 19.78% 높은 AUC-ROC를 보고했다. XDAC은 뉴스 댓글처럼 **짧고 비격식적인 텍스트**를 겨냥해 1.3M 인간 댓글과 1M LLM 댓글, 총 14개 모델을 사용한 대규모 벤치마크를 구축했고, 댓글 탐지에서 98.5% F1, 생성 모델 귀속에서 84.3% F1을 보고했다. 또 최근 Findings of EACL 논문은 한국어 전용 비지도 프레임워크를 제안해 F1 0.963, ROC-AUC 0.985를 보고하며, 저자 주석 데이터 없이도 강한 성능이 가능함을 보였다.

**After**

> 한국어 쪽은 최근에야 쌓이기 시작했다. KatFish 는 에세이 / 시 / 논문 초록 세 장르에서 인간 / LLM 텍스트를 모은 첫 한국어 탐지 벤치마크다. 한국어 언어 특징에 기반한 KatFishNet 은 기존 방법보다 평균 19.78 % 높은 AUC-ROC 를 냈다. XDAC 은 뉴스 댓글처럼 **짧고 비격식적인 텍스트**를 겨냥해 1.3 M 인간 댓글 + 1 M LLM 댓글 + 14 개 모델로 큰 벤치마크를 만들었다. 댓글 탐지 98.5 % F1, 생성 모델 귀속 84.3 % F1. Findings of EACL 한국어 전용 비지도 프레임워크는 F1 0.963 / ROC-AUC 0.985 로, 저자 주석 데이터 없이도 강한 성능이 가능함을 보였다.

**핵심 변화**

- "의미 있는 축적이 나타났다" → "쌓이기 시작했다" (#3 번역체 + 명사화 → 서술형)

## 사례 3. "Humanizer 설계 원칙" 단락

**Before**

> 이 설계는 두 방향의 증거를 따른다. 하나는 탐지 연구가 보여 준 **AI 텍스트의 한국어적 편향**이고, 다른 하나는 StyleKQC가 보여 준 **한국어 스타일 전환이 공손성, 화행, 형태소 수준 표현을 함께 움직여야 자연스럽다**는 점이다. Anthropic의 권고 역시 원하는 출력 포맷을 정확히 지정하고, 예시를 제공하고, 복잡한 작업은 연쇄형 서브태스크로 나누는 편이 일관성을 높인다고 설명한다. 따라서 Humanizer는 프롬프트 하나에 전부 맡기기보다, **규칙 기반 전처리 + 모델 기반 재작성 + 사실 보존 검사**의 3단 구조가 바람직하다.

**After**

> 설계 근거는 둘이다. 하나는 탐지 연구가 보여 준 **AI 텍스트의 한국어적 편향**, 다른 하나는 StyleKQC 가 보여 준 **한국어 스타일 전환이 공손성·화행·형태소 표현이 같이 움직여야 자연스럽다는 점**. Anthropic 권고도 출력 포맷을 분명히 지정하고, 예시를 주고, 복잡한 작업은 서브태스크로 쪼개라고 한다. Humanizer 도 프롬프트 하나에 다 맡기기보다 **규칙 기반 전처리 + 모델 기반 재작성 + 사실 보존 검사** 3 단 구조가 낫다.

**적용 카테고리**

- "이 설계는 두 방향의 증거를 따른다" → "설계 근거는 둘이다" (#1 군더더기 / #11 명사 편중)
- "~이고, 다른 하나는 ~ 점이다" 의 종결 군더더기 단축 (#11)
- "~의 권고 역시 ~ 설명한다" → "~ 권고도 ~ 한다" (#3 ~의 남용 / #7 역시 삭제)
- "따라서 Humanizer 는 ~ 바람직하다" → "Humanizer 도 ~ 낫다" (#7 따라서 / #1 바람직하다)

**보존**: StyleKQC, Anthropic, "AI 텍스트의 한국어적 편향", "공손성·화행·형태소 표현", "규칙 기반 전처리 + 모델 기반 재작성 + 사실 보존 검사" 등 핵심 개념·인용은 그대로.

## 사례 4. "한계" 1 + 2 단락

**Before**

> 첫째, 어떤 특징도 단독으로는 결정적이지 않다. Korean-specific features가 강한 것은 사실이지만, MAGE와 RAID가 보여 주듯 도메인이 바뀌고 생성기가 달라지면 성능은 흔들린다. 공격적 재작성은 탐지 신호를 약화시키기도 한다. 따라서 이 가이드는 "증거의 방향"을 제시할 뿐, 보편적 지문을 약속하지 않는다.
>
> 둘째, 사람 판단도 안정적이지 않다. 한국어 참가자 실험에서 정답률은 낮았고, LREAD처럼 루브릭과 캘리브레이션이 있어야 합의도가 올라간다. 따라서 "전문가 직감"은 필요하지만 충분조건이 아니다.

**After**

> 첫째, 단독으로 결정적인 특징은 없다. Korean-specific features 가 강한 건 사실이지만, MAGE 와 RAID 가 보여 주듯 도메인 / 생성기가 바뀌면 성능이 흔들린다. 공격적 재작성은 탐지 신호를 약하게 만든다. 이 가이드는 "증거의 방향" 만 줄 뿐, 보편적 지문을 약속하지 않는다.
>
> 둘째, 사람 판단도 안정적이지 않다. 한국어 참가자 실험에서 정답률이 낮았고, LREAD 처럼 루브릭과 캘리브레이션이 있어야 합의도가 오른다. "전문가 직감" 은 필요하지만 충분조건은 아니다.

**적용 카테고리**

- "어떤 특징도 단독으로는 결정적이지 않다" → "단독으로 결정적인 특징은 없다" (#3 번역체)
- "강한 것은 사실이지만" → "강한 건 사실이지만" (구어 회복)
- "도메인이 바뀌고 생성기가 달라지면" → "도메인 / 생성기가 바뀌면" (#7 절 단축)
- "약화시키기도 한다" → "약하게 만든다" (#3 ~화 / #6 수동 분위기)
- "따라서 이 가이드는 ~ 제시할 뿐" → "이 가이드는 ~ 만 줄 뿐" (#7 따라서 / #1 제시)
- "정답률은 낮았고" → "정답률이 낮았고" (조사 자연화)
- "올라간다" → "오른다" (#1 군더더기)
- 마지막 문장 "따라서" 삭제 (#7)

**보존**: MAGE, RAID, LREAD, "Korean-specific features", "증거의 방향" 등 인용·개념. 한계 4 가지 구조도 보존.

## 사례 5. 부록용 AI-유사 vs 자연화 예시 단락

이 단락은 **메타** 다 — 위키가 직접 든 humanize 예시 자체를 평가한다. 위키는 두 예시를 나란히 배치하고 그 차이를 설명하는 한 단락을 덧붙인다. 그 한 단락이 어떻게 humanize 됐는지 본다.

**Before**

> 첫 번째 문장은 안전한 평가형 표현과 연결어가 과밀하고, 두 번째 문장은 명사구를 서술형으로 풀고 연결어를 줄여 한국어 리듬을 살린다. 이 차이는 KatFish가 지적한 명사 편중·낮은 서술성·정형화된 흐름과 맞물린다.

**After**

> 첫 번째 문장은 안전 평가 표현과 연결어가 빽빽하고, 두 번째 문장은 명사구를 서술형으로 풀고 연결어를 줄여 한국어 리듬을 살린다. KatFish 가 짚은 명사 편중·낮은 서술성·정형화된 흐름과 맞물린다.

**적용 카테고리**

- "안전한 평가형 표현" → "안전 평가 표현" (명사형 압축, #2 공허한 형용사 정리)
- "과밀하고" → "빽빽하고" (#1 격식 어휘 → 자연어)
- "이 차이는 ~ 지적한 ~ 와 맞물린다" → "KatFish 가 짚은 ~ 와 맞물린다" (#4 "이 ~" 접두사 / #1 "지적한")

이 사례는 **얼마나 적게 손대도 되는지**를 보여 준다. 단락이 짧고 이미 비교적 정돈된 글이라, 12 카테고리 중 3 개만 적용해도 충분하다.

## Humanizer 가 *덜* 손댄 부분

의도적으로 손대지 않은 곳들. **자연스러움보다 정확성이 우선** 인 영역이다.

1. **Feature schema 표** — "한국어적 이유 / 의심 방향 / 비고" 컬럼은 카탈로그성 정보. 압축하면 의미가 무너진다.
2. **부록 References 섹션** — 논문 제목 / 저자 / 연도 / URL 모두 fact.
3. **코드 블록 (Python / Kotlin / Java / YAML)** — 공백 한 칸이라도 바꾸면 문법 깨짐.
4. **Mermaid flowchart** — 노드 라벨이 humanize 대상으로 보일 수 있지만, 짧은 다이어그램 라벨은 "자연어" 가 의미가 약하고 잘못 만지면 가독성 손해.
5. **Executive Summary 의 KatFish / XDAC 인용 수치** — 위 사례 2 처럼 보존.

## 한계와 주의

이 비교 작업에서 humanizer 가 약했던 지점들.

1. **학술적 정확성 vs 구어성 트레이드오프**. 위키는 학술 / 기술 위키 톤이라 강한 캐주얼화는 부적합. 그래서 발화체 변환은 거의 쓰지 않고, 주로 #1 / #3 / #6 / #7 / #11 만 동작했다.
2. **외래어·전문 용어 보존 문제**. "AUC-ROC", "lexical diversity", "POS n-gram" 같은 용어는 한글로 풀면 정보 손실. 그대로 두는 게 정답.
3. **표·표 캡션의 두 톤 충돌**. 표 캡션 prose 는 humanize 했는데 표 본문은 손대지 않으니, 한 섹션 안에 두 톤이 공존한다. 학술 위키에서는 허용 가능하지만 일반 글에는 권장하지 않음.
4. **반복 인용 단락**. References 가 본문 안에 자주 등장하면 humanizer 가 인용을 잘못 건드릴 위험이 있다 — 이번엔 위키 본문에서 인용 마커를 미리 제거(`citeturn` 류 정리) 한 뒤 humanize 해서 안전했다.
