# 한국어 AI 글쓰기 "AI 티" 체크리스트

> Korean AI writing tells — 입력 텍스트에서 아래 패턴이 보이면 더 자연스러운 대안으로 바꾼다.

**과교정 방지 룰**:
- 한 문단에 **3곳 이상**은 건드리지 않는다.
- **전체 문장 수의 20%** 이상은 수정하지 않는다(예: 20문장 글이면 최대 4문장만 손댄다).
- 팩트 / 숫자 / 고유명사 / 인용문 / 링크는 **절대** 변경하지 않는다.
- 살짝 덜 매끄러운 게 더 사람답다. 과도한 세련미는 오히려 AI 티.

---

## 1. 강조어 남발 (Empty Intensifiers)

AI는 모든 게 "매우 / 정말 / 굉장히 / 획기적"이다. 실제로는 대부분 평범하다.

| 나쁨 | 자연스러움 |
|---|---|
| 매우 중요한 | 중요한 / 꽤 중요한 |
| 정말 효과적인 | 효과적인 / 실제로 도움되는 |
| 굉장히 획기적인 | 새로운 / 눈에 띄는 |
| 매우 유용한 | 쓸만한 / 실제로 쓰이는 |
| 엄청난 | 큰 / 꽤 / (구체 수치로) |
| 한층 더 | 더 / (삭제) |
| 더더욱 | 더 / 훨씬 |
| 보다 ~한 | 더 ~한 ("보다"는 비교 대상이 명시될 때만) |
| 너무나도 | 너무 / (삭제) |

## 2. 공허한 형용사 (Empty Adjectives)

구체성이 없는 꾸밈말. 빼거나 구체로 바꾼다.

| 나쁨 | 자연스러움 |
|---|---|
| 다양한 방법 | 몇 가지 방법 / 세 가지 방법 |
| 폭넓은 경험 | 긴 경험 / 5년 이상 경험 |
| 혁신적인 솔루션 | 새 방식 / (구체 기능 명시) |
| 효과적인 전략 | 실제로 먹히는 전략 / (성과 수치) |
| 체계적인 접근 | 순서대로 / 단계별로 |
| 광범위한 분야 | 여러 분야 / (구체 영역 나열) |
| 포괄적인 가이드 | 전체 가이드 / 처음부터 끝까지 |
| 종합적인 분석 | 전체 분석 / 모아서 본 분석 |
| 전반적인 흐름 | 전체 흐름 / 큰 흐름 |
| 특별한 의미 | (구체) / 남다른 의미 |
| 의미 있는 변화 | (어떤 변화인지 구체) |
| 가치 있는 시도 | (어떤 가치인지 구체) |

## 3. 과격식 / 번역체 (Translation-ese & Stiff Formalism)

영어 / 일본어 직역 흔적, 격식체 남용.

| 나쁨 | 자연스러움 |
|---|---|
| ~에 대해 말씀드리자면 | ~에 대해 얘기하면 / (바로 본론) |
| 다음과 같습니다 | (바로 내용으로) |
| 저는 ~입니다. 저는 ~합니다. | (주어 생략) |
| ~라는 것입니다 | ~다 / ~이다 |
| ~에 있어서 | ~에서 / ~할 때 |
| ~함에 있어 | ~할 때 |
| ~에 다름 아니다 | ~다 / ~일 뿐이다 |
| ~에 의해 | ~가 / ~로 |
| ~을 통해 | ~로 / ~으로 |
| ~에 대한 (남용) | ~의 / (생략) |
| 일컬어 ~라 한다 | ~라고 한다 |
| 이른바 | (대부분 삭제) |

## 4. "것이다" / "이러한" / "해당" 과다

영어 *such / these / the said* 직역과 AI 기본 마침법 "~인 것이다"가 한 문단에 여러 번 나오면 거의 확실한 AI 티.

| 나쁨 | 자연스러움 |
|---|---|
| 사실 ~인 것이다 | 사실 ~예요 / ~다 |
| 문제인 것이다 | 문제다 / 문제예요 |
| 해야 하는 것이다 | 해야 한다 |
| 이러한 이유로 | 이 때문에 / 그래서 |
| 이러한 상황에서 | 이런 상황에선 |
| 이러한 사실은 | 이 사실은 |
| 이러한 방식으로 | 이 방식으로 / 이렇게 |
| 해당 기능은 | 그 기능은 |
| 해당 시스템 | 그 시스템 |
| 해당 사항은 | 이 사항은 / 그건 |

## 5. 판에 박힌 개시 / 마무리

### 개시 문구 (대개 삭제가 정답)

- "안녕하세요! 오늘은~" → (바로 본론)
- "다들 궁금하셨을 텐데요~" → (삭제)
- "많은 분들이 ~하시는데요" → (자기 관찰로 — "나도 ~", "주위에서 ~")
- "~에 대해 알아보겠습니다" → (바로 내용)
- "~에 대해 살펴보도록 하겠습니다" → (바로 내용)
- "오늘 이 글에서는" → (삭제 / 한 줄 요약)
- "본격적으로 ~를 시작하기 전에" → (대부분 삭제)

### 마무리 문구

- "도움이 되셨길 바랍니다" → (삭제 또는 구체 후속)
- "궁금한 점이 있으시면 언제든지" → (삭제)
- "감사합니다!" (무조건 붙이기) → 상황에 맞을 때만
- "함께 ~해봅시다!" (강제 호응) → (삭제)
- "여러분의 ~을 응원합니다" → (대부분 삭제)
- "이상으로 ~을 마치겠습니다" → (그냥 끝)

## 6. 수동태 / 모호한 주어 (Passive / Hidden Agents)

AI는 행위 주체를 숨기는 수동태로 도망간다.

| 나쁨 | 자연스러움 |
|---|---|
| ~이 될 수 있습니다 | ~할 수 있다 |
| ~되어집니다 (이중 수동) | ~된다 |
| 고려되어야 합니다 | 고려해야 한다 / 챙겨야 한다 |
| 요구됩니다 | 필요하다 / ~해야 한다 |
| 활용되어 왔다 | 써왔다 / 자주 쓰였다 |
| 평가되고 있다 | (누가) 평가한다 |
| 인식되어 있다 | 알려져 있다 / (누가) 안다 |
| 진행되어 왔다 | 해왔다 |
| 주어진다 | (누가) 준다 |
| 발견된다 | 보인다 / 나온다 |

## 7. 불필요한 연결어 (Filler Connectives)

AI는 문장마다 "또한 / 뿐만 아니라 / 더불어 / 따라서"로 잇는다. 대부분 없어도 된다.

| 나쁨 | 자연스러움 |
|---|---|
| 또한 ~할 수 있습니다 | (새 문장으로 시작, 연결어 삭제) |
| 뿐만 아니라 | (대부분 삭제) |
| 더불어 | (대부분 삭제) |
| 나아가 | (대부분 삭제) |
| 한편 | 진짜 대조할 때만 유지 |
| 따라서 | 그래서 / (삭제) |
| 그러므로 | 그래서 / 그러니까 |
| 결론적으로 | 결국 / (삭제) |
| 즉 / 다시 말해 | (한 번만 / 삭제) |
| 아울러 | 그리고 / (삭제) |
| 한마디로 | (대부분 삭제) |
| 요컨대 | 정리하면 / (삭제) |

## 8. 과도한 parallelism / 3항 나열 강제

AI는 항목을 셋으로 맞추고 문장 길이를 균등하게 만든다. 의도적으로 흐트린다.

**나쁨**:
> 빠르고 정확하며 효율적입니다. 간단하고 명확하며 실용적입니다.

**자연스러움**:
> 빠르고 정확하다. 쓰기도 어렵지 않다.

| 나쁨 | 자연스러움 |
|---|---|
| 기술, 인재, 자금의 세 가지 요소 | 기술이 필요해. 인력도, 돈도. |
| 첫째, ~. 둘째, ~. 셋째, ~. | 우선 ~. 그리고 ~. 마지막으로 ~. |
| 안전하고 빠르고 저렴하다 | 안전하고 빠르다. 가격도 괜찮다. |
| 학습, 적응, 성장 | 배우면서 적응하고 자란다 |

## 9. 존댓말 레벨 불일치 (Honorific Mismatch)

같은 글 안에서 "합니다체"와 "해요체"를 섞거나, 전체 톤과 안 맞는 과격식을 넣는다.

- 캐주얼 포스트에 "~하시기 바랍니다" → "~해보세요" / "~하면 좋아요"
- 카톡/DM 톤에 "~하십니다" → "~해요" / "~합니다" 중 하나로 통일
- 친근한 인사 후 갑자기 "~하옵니다" → 톤 통일
- "하시는 분들께서는" → "하는 분들은" / "하시는 분들은"
- "가능하시다면" → "가능하면" / "되면"

### 9-A. 종결어미 톤 보존 (humanize 시 최우선 룰)

humanize 결과는 **raw의 종결어미 톤을 도메인 디폴트와 함께 결정**한다. "짧고 단단하게"를 핑계로 ~다체 글말체로 통일하지 않는다.

**도메인별 종결어미 디폴트**:

| 도메인 | 디폴트 어미 | ~다체 글말체 |
|---|---|---|
| 발화체 (YouTube / 팟캐스트 / 강의 / 라이브 인사말) | ~합니다체 / ~해요체 | ❌ **사용 금지** |
| 격식 글 (이메일 / 공문 / 비즈니스 메모) | ~합니다체 / ~드립니다 | ❌ raw 톤 strict 보존 |
| 캐주얼 (DM / 메신저 / 짧은 SNS) | raw 톤 그대로 (~해요체 / 반말) | ❌ raw 톤 보존 |
| 글말체 (블로그 / 에세이 / 책 / 마케팅 카피) | ~다 / ~한다 (글말체) | ✓ raw가 ~합니다체여도 ~다체 OK (블로그 표준) |

**한 글 내 어미 통일**: 어떤 도메인이든 한 글에서 어미가 섞이지 않게 한다 — raw에서 우세한 어미 군으로 통일한다.

**금기 패턴 (반드시 catch)**:
- 발화체 raw가 "~알아보겠습니다 / ~함께해 주세요 / ~거예요" 인데 humanized가 "~정리해본다 / ~팁이다 / ~달라진다" 로 가는 것 — 이건 카테고리 #9의 정면 위반이다.
- humanized 첫 문장이 ~합니다체 + 본문이 ~다체 + 마지막이 ~합니다체로 회귀하는 패턴 — 한 글 안에 두 어미군 동시 존재.

**예시**:

| Raw (발화체 YouTube) | ❌ 잘못된 humanize | ✓ 올바른 humanize |
|---|---|---|
| "오늘은 ~ 알아보겠습니다" | "오늘은 ~ 정리해본다" | "오늘은 ~ 정리해볼게요" |
| "~수 있을 거예요" | "~달라진다" | "~달라져요" |
| "그럼 시작하겠습니다!" | "시작합니다." (← 다른 문장은 ~다체인데 여기만 ~합니다체) | "시작할게요." |

## 10. 이모지 / 이모티콘 남발

포스트당 1~2개, 의미 있는 자리에만. AI는 섹션 헤더마다 🚀💡✨을 박는 경향.

- 섹션 헤더 / 불릿 앞 이모지 무한 반복 → 최대 1~2개만
- 문장 중간 이모지 → 대부분 제거
- 이모지로 강조 (🔥🎯) → 단어로 표현
- 마무리에 손/박수 이모지 (👏🙌) 자동 첨부 → 대부분 삭제

## 11. 단정 회피 / 과한 hedging

자기 글에 자신이 없을 때 AI가 자주 쓰는 방어막.

| 나쁨 | 자연스러움 |
|---|---|
| ~일 수도 있습니다 (남발) | (아는 건 단언) |
| ~라고 할 수 있습니다 / ~라고 할 수 있다 | ~다 / ~이다 (단언) |
| ~인 것 같습니다 (남발) | ~다 / ~예요 |
| 아마도 / 어쩌면 (반복) | (한 번만 살리기) |
| 제 생각에는 (과다) | (신념이 분명한 문장은 그냥 서술) |
| 어느 정도는 | (대부분 삭제) |
| 일반적으로는 | (대부분 삭제 / 구체 사례) |
| 대부분의 경우 | 보통 / 거의 (간결화) |
| 여러 가지 측면에서 | (구체 측면 나열) |

## 12. AI 고빈도 어휘 (Korean LLM Vocabulary Tells)

2023년 이후 한국어 LLM 텍스트에서 유난히 자주 등장하는 단어들. 영어 *delve / tapestry / leverage* 의 한국어 짝.

| 나쁨 | 자연스러움 |
|---|---|
| 활용 / 활용하여 | 사용 / 쓰기 / ~로 |
| ~을 활용한 접근 | ~로 하는 방식 / ~ 방법 |
| 시사한다 | 보여준다 / 의미한다 |
| 극대화 | 최대로 / 가장 크게 |
| 강화 | 더 단단하게 / 키우다 |
| 통합 | 합치기 / 묶기 |
| 지속 (~을 지속하다) | 계속 / 꾸준히 |
| ~의 관점에서 | ~로 보면 / (대부분 삭제) |
| 극복 | 이겨내기 / 풀기 |
| 도모 | 챙기기 / 만들기 |
| 도출 | 뽑아내기 / 끌어내기 |
| 모색 | 찾기 / 살펴보기 |
| 함의 | 뜻 / 속뜻 |
| 저변 | 바탕 / (삭제) |
| ~의 일환으로 | (대부분 삭제) |

## 부록: 적용 순서

1. **개인 금지어/선호어 리스트**가 있으면 먼저 적용 (`examples/personal-list.md` 참고).
2. 위 12 카테고리를 차례로 스캔.
3. 한 문단에 3곳 이상 건드렸으면 멈추고 다음 문단으로.
4. **전체 문장 수의 20%**를 넘으면 영향이 작은 변경부터 되돌리기.
5. 출력 시 **다듬어진 전체 글** + **주요 변경 (최대 5개)** 형식. 변경이 5개 초과면 "+N개 더 — '전체 diff' 요청 시 모두 표시" 한 줄 추가.

## 부록 A. 연구 근거 (Research Findings)

이 카탈로그는 단순 관찰뿐 아니라 한국어 AI 탐지 연구의 정량 보고치와도 결이 맞다. 핵심 수치만 요약한다 — 전체 근거 / 평가 프로토콜 / 윤리 한계는 별도 위키 문서 [`korean-humanizer-research.md`](../korean-humanizer-research.md) 참조.

### A.1 KatFish — 쉼표 사용 패턴 (인간 vs LLM)

KatFish (2025) 가 한국어 에세이 / 시 / 논문 초록에서 보고한 쉼표 사용 차이. **카테고리 #7 (불필요한 연결어)** 와 같은 결을 가진 정량 근거.

| 장르 | 지표 | 인간 | LLM |
|---|---:|---:|---:|
| 에세이 | 쉼표 포함 문장 비율 | 26.31 | 61.03 |
| 에세이 | 평균 분절 길이 | 4.35 | 8.56 |
| 논문 초록 | 쉼표 포함 문장 비율 | 47.48 | 65.21 |
| 논문 초록 | 평균 분절 길이 | 9.07 | 11.55 |

→ 한국어 LLM 텍스트는 **쉼표를 더 자주 / 더 뒤에 / 더 긴 분절로** 쓴다. humanize 시 긴 쉼표 분절은 두 문장으로 쪼개는 편이 한국어답다.

### A.2 KatFish — 한국어 전용 punctuation 특징의 강도

| 장르 | 장르별 최고 baseline AUC | KatFish punctuation AUC |
|---|---:|---:|
| 에세이 | 81.27 | **94.88** |
| 시 | 66.02 | **73.10** |
| 논문 초록 | 57.33 | **75.62** |

→ 한국어에서 쉼표·종결어미·POS 다양성 같은 **언어 고유 특징**이 영어 모델식 일반 baseline 보다 평균 +19.78% 높은 AUC-ROC 를 보인다.

### A.3 XDAC — 짧은 비격식 텍스트의 별도 신호

뉴스 댓글처럼 짧은 텍스트에서는 **반복 자모 / 감정 기호 / 특수문자 / 비격식 표현** 의 결핍이 강한 신호다 (XDAC 댓글 탐지 98.5 % F1).

→ humanize 출력이 댓글 / 채팅 도메인일 경우 모든 매끈함을 다 깎아내면 오히려 비현실적이다. 카테고리 #9-A (종결어미 톤 보존) 와 결합해 도메인 디폴트로 판단한다.

### A.4 사람 판정의 한계 (LREAD / 사용자 실험)

- 한국어 인간 vs AI 에세이 식별 실험: 116 명 참가자 정답률 54.8 ~ 60.6 % (chance 근방).
- LREAD: 직관 정확도 0.60 → 루브릭 적용 후 0.90, 평가자 간 κ -0.09 → 0.82.

→ "사람이 보면 다 안다" 는 **거짓**이고, 동시에 자동 점수 하나로 사람을 판정해서도 안 된다는 뜻. humanizer 도 평가도 **다중 신호 + 사람 검토** 가 정답.

## 부록 B. 한국어 전용 feature schema (탐지·평가 보조용)

humanizer 는 *편집기*지만, 같은 신호를 *탐지/평가*에도 쓸 수 있다. 평가 자동화 / 회귀 테스트 만들 때 참조.

| 특징 | 측정 | 의심 방향 |
|---|---|---|
| 쉼표 포함 비율 | 쉼표 포함 문장 / 전체 문장 | 높을수록 LLM 의심 |
| 쉼표 평균 상대 위치 | 쉼표 위치 / 문장 길이 | 뒤로 갈수록 의심 |
| 쉼표 분절 길이 | 쉼표로 분리된 세그먼트 평균 길이 | 길수록 의심 |
| 명사 비율 | 명사 형태소 / 전체 형태소 | 높을수록 의심 (술어·어미 비율과 함께) |
| 술어·어미 비율 | 용언+어미 / 전체 형태소 | 낮을수록 의심 |
| 종결어미 엔트로피 | 문장말 어미 분포의 엔트로피 | 너무 낮으면 의심 |
| 어휘 다양성 (MATTR) | 형태소 기반 type/token | 낮을수록 의심 |
| 연결어 빈도 | "또한 / 따라서 / 즉 / 한편 / 뿐만 아니라" 빈도 | 높을수록 의심 |
| 비격식 표지 지수 | "ㅋㅋ / ㅎㅎ / ㅠㅠ / ?! / ..." 빈도 | 댓글·채팅에서 너무 낮으면 의심 |
| 종결어미 도메인 적합도 | 카테고리 #9-A 표 기준 | 발화체에 ~다체 글말체면 fail |

판정은 절대 임계가 아니라 **장르-조건부 z-score** 기준. 단일 신호로 판단하지 않고 상관이 낮은 신호 3 개 이상이 같은 방향이면 "검토 필요" 로만 올린다 (자동 단정 금지).

## 부록 C. 사람 평가용 한국어 루브릭 (LREAD 기반)

humanize 결과를 사람이 평가할 때 쓰는 5점 루브릭. README 의 Full Example, `examples/agent-vs-skill.md` 의 정성 평가가 이 축을 따른다.

| 축 | 1점 | 5점 |
|---|---|---|
| 문장 흐름 | 기계적 단조로움 | 자연스러운 리듬 |
| 레지스터 | 과정렬 / 혼재 (예: 발화체에 ~다체) | 도메인 디폴트와 일치 |
| 구체성 | 추상적 일반론 | 맥락적 구체성 |
| 감정 / 입장 | 평평함 | 미세한 변화 살아 있음 |
| 비격식 표지 | 과도한 매끈함 | 장르 적합한 비격식 표지 |

## 부록 D. 윤리·한계

- 이 카탈로그는 **편집 보조 / 연구용 분석** 목적이지, 사람 글을 자동으로 "AI 가 썼다" 고 판정하는 도구가 아니다.
- 단순한 문체, 학습자 문체, 짧은 글은 단일 신호로 보면 AI 로 오탐될 수 있다 (영어권 비원어민 오탐과 같은 결).
- humanize 는 **사실 보존 + 출처 보존** 우선. 무근거 세부 추가 / 사칭 / 평가 우회 목적 사용은 지원하지 않는다.
- 법률 / 의료 / 보안 / 재무 텍스트는 비격식 변환 / 생략을 원칙적으로 금지 (SKILL.md / PROMPT.md 의 절대 규칙과 동일).
